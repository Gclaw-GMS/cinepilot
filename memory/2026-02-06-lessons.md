# Mission Control Build - Lessons Learned

## Date: 2026-02-06
## Project: Mission Control for OpenClaw

### What Went Well
- Used Ralph Loops methodology effectively - state management in files worked great
- Stayed within budget ($2.45 spent of $5.00)
- Completed full stack: Express backend + React frontend + Supabase + OpenClaw integration
- Git commits were clean and descriptive
- Regular check-ins kept user informed

### What Went Wrong / Lessons
1. **Screenshots failed because dev server wasn't running**
   - Started dev server, then it stopped before screenshots were taken
   - SOLUTION: Always verify server is running before taking screenshots
   - SOLUTION: Use `nohup` or background process to keep server alive
   - SOLUTION: Check server health before screenshot commands

2. **Browser automation flaky**
   - Chrome not found issues
   - Playwright/puppeteer installation problems
   - SOLUTION: Use existing browser (Brave detected) via CDP
   - SOLUTION: Keep screenshot tools simple (wget/curl for HTML, not full browsers)

3. **npm permission issues**
   - Cache folder ownership problems
   - SOLUTION: Use `--cache /tmp/npm-cache` flag always
   - SOLUTION: Install locally not globally when possible

### Technical Wins
- Heartbeat orchestrator with staggered cron worked perfectly
- Real-time Supabase subscriptions implemented correctly
- Dark mode toggle with class-based Tailwind approach
- Zustand stores structured well

### For Next Time
- Always verify services are running before dependent operations
- Test screenshot flow end-to-end before claiming completion
- Keep alternative screenshot methods ready (CDP, canvas, etc.)
- Monitor background processes better
